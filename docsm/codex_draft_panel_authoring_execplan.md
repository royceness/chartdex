## Purpose / Big Picture

After this change, Codex can create draft personal dashboards and panels for a ChartDex user. A user can ask Codex to investigate a metric movement, then ask it to create a focused draft panel showing the segment or metric behind the finding. Codex still does not receive browser credentials, raw SQL access, or broad dashboard mutation. It receives dynamic tools that validate a structured panel specification and write only personal draft content owned by the current authenticated user.

The user-visible proof is: after Codex calls the draft authoring tool, `GET /api/dashboards` includes a personal draft dashboard, and `GET /api/dashboards/{id}` returns authored panels with data evaluated through the same bounded metrics provider used by the rest of ChartDex.

## Decision Log

- Decision: Limit Codex writes to personal draft dashboards and draft panels owned by the authenticated user.
  Rationale: This gives the demo authoring moment while avoiding org-wide mutations and preserving user review before sharing or publishing.
  Date/Author: 2026-05-18 / Codex

- Decision: Store authored panels as structured query JSON, not SQL.
  Rationale: The existing metrics provider already enforces allowlisted metrics, dimensions, filters, date ranges, and limits. Reusing that keeps authoring within the same secure query boundary.
  Date/Author: 2026-05-18 / Codex

- Decision: Add a validation tool separate from create tools.
  Rationale: Codex benefits from a repair loop. A validation tool can return precise field-level errors and examples before any write occurs.
  Date/Author: 2026-05-18 / Codex

## Context and Orientation

The backend lives in `backend/app`. `backend/app/codex_tools.py` defines dynamic tools. These are schemas sent to `codex space app-server`; when Codex calls a tool, the backend handles it with a server-side `ChartDexToolContext` containing `app_db_path`, `org_id`, `user_id`, and `thread_id`.

The existing metrics query capability is in `backend/app/metrics_provider.py` as `SQLiteMetricsProvider.query_metrics(query)`. It accepts structured query dictionaries, not SQL. The dashboard API is in `backend/app/main.py`, and app-state database helpers live in `backend/app/database.py`.

The frontend lives in `frontend/src`. It lists dashboards with `fetchDashboards()` and details with `fetchDashboardDetail()`. It polls Codex threads while a thread is queued or running, but currently does not refresh dashboard lists/details during polling.

## Plan of Work

Add `dashboard_panels` app-state storage for authored panels. Add optional columns to `dashboards` for `status`, `created_by`, and `source_thread_id`, with idempotent schema migration for existing app-state databases. Draft dashboards use `space = personal`, `owner_user_id = auth.user_id`, `status = draft`, and `created_by = codex`.

Create `backend/app/dashboard_authoring.py` to hold validation and creation logic. Define allowed panel types `line` and `bar`, value formats `currency`, `percent`, and `integer`, one metric per authored panel, and a limited query shape matching `query_metrics`. `validate_panel_spec` returns field-level errors instead of raising where possible. `create_draft_dashboard` and `create_draft_panel` call the same validator before writing.

Update `backend/app/database.py` so `get_dashboard_detail` combines provider panels and authored panels. For authored draft dashboards, provider panels are empty and authored panels supply all visible panels. For existing personal or org dashboards, authored panels append after provider panels if the authenticated user owns them. The renderer evaluates authored panel `query_json` through `provider.query_metrics` and converts rows into the frontend's existing `MetricPoint` or `CategoryPoint` shapes.

Update `backend/app/codex_tools.py` with `get_authoring_capabilities`, `validate_panel_spec`, `create_draft_dashboard`, and `create_draft_panel`.

Update frontend polling so when active Codex work finishes, the dashboard list/details refresh. This lets a Codex-created draft dashboard or panel appear after the thread completes.
